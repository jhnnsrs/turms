# Input Funcs

The `input_funcs` plugin generates a small **factory function for every GraphQL
input type** that builds and returns the corresponding pydantic input model.

It is modeled on the [`funcs`](funcs) plugin, but instead of building a
`variables` dict and calling an executor, it constructs the model directly
(`return SomeInput(**data)`). It follows the same
[UNSET / "backend owns defaults"](../migration-defaults-unset) model: optional
fields default to the `UNSET` sentinel and are only included when the caller
provides them.

## Why use it?

Constructing nested input models by hand is verbose and easy to get wrong
(aliases, optionality, nested inputs). The factory gives you a fully-typed,
documented function with required fields as positional arguments and optional
fields as keyword arguments:

```python
# Generated for:
#   input UserInput { name: String!  age: Int  address: AddressInput  tags: [String!] }
#   input AddressInput { street: String!  zip: String }

def user_input(
    name: str,
    age: Union[Optional[int], UnsetType] = UNSET,
    address: Union[Optional[AddressInput], UnsetType] = UNSET,
    tags: Union[Optional[Iterable[str]], UnsetType] = UNSET,
) -> UserInput:
    "Creates a UserInput ..."
    data: Dict[str, Any] = {}
    data['name'] = name
    if age is not UNSET:
        data['age'] = age
    if address is not UNSET:
        data['address'] = address
    if tags is not UNSET:
        data['tags'] = tags
    return UserInput(**data)
```

Usage:

```python
addr = address_input(street="Main", zip="12345")
user = user_input(name="bob", age=30, address=addr, tags=["x", "y"])

# Omitted optionals stay unset, so exclude_unset drops them entirely:
user_input(name="alice").dict(by_alias=True, exclude_unset=True)
# -> {"name": "alice"}
```

## Ordering

The plugin needs the model classes generated by the `inputs` plugin (and any enum
classes), so it **must run after** `InputsPlugin`. Input types that were not
generated (e.g. skipped by config) are silently skipped.

```yaml
plugins:
  - type: turms.plugins.enums.EnumsPlugin
  - type: turms.plugins.inputs.InputsPlugin
  - type: turms.plugins.input_funcs.InputFuncsPlugin
```

## Configuration

| Option | Type | Default | Description |
| --- | --- | --- | --- |
| `coercible_scalars` | `Dict[str, str]` | `{}` | Map scalar → a friendlier python type for the factory params (e.g. `ID: pydantic.UUID4`); the model coerces. Merges on top of the global `coercible_scalars`, overriding per-scalar. |
| `skip_underscore` | `bool` | `true` | Skip input types whose name starts with `_`. |
| `skip_unreferenced` | `bool` | `true` | Only generate factories for input types referenced by your documents. |
| `prepend` | `str` | `""` | Prefix for the generated factory function name (e.g. `make_` → `make_user_input`). |
| `extract_documentation` | `bool` | `true` | Generate a docstring from the input type / field descriptions. |

```yaml
- type: turms.plugins.input_funcs.InputFuncsPlugin
  prepend: "make_"
  coercible_scalars:
    ID: pydantic.UUID4
```

:::tip
Use a snake_case styler so the factory function name (`user_input`) differs from
the PascalCase model class (`UserInput`); otherwise turms appends a trailing `_`
to avoid shadowing the class.
:::

## Coercible scalars

`coercible_scalars` lets callers pass a friendlier python type for a scalar
parameter while the model performs the coercion. It can be set globally (on the
[generator config](../config)) and overridden per-plugin:

```yaml
extensions:
  turms:
    coercible_scalars:      # global base
      ID: pydantic.UUID4
    plugins:
      - type: turms.plugins.input_funcs.InputFuncsPlugin
        coercible_scalars:  # plugin override wins for String
          String: pathlib.Path
```
